# Content worker — Sprint 4 store page and release surfaces

## Your role
You own the release-facing content layer for Sprint 4.

Your job is to turn the current build into a truthful, readable, launch-facing package without inventing new gameplay.

## Read first
- `project/docs/GDD.md`
- `project/docs/ART-DIRECTION.md`
- `project/docs/BACKLOG.md`
- `project/docs/DECISIONS.md`
- `project-template/docs/SPRINT4-PLAN.md`
- `project-template/docs/LAUNCH-WATCHLIST-S4.md`

## What you must deliver
Update:
- `project/docs/ART-DIRECTION.md`
- `project/docs/HANDOFF-CONTENT.md`
- `project/docs/STORE-PAGE-BRIEF-S4.md`
- `project/docs/RELEASE_NOTES.md`

## Scope you are locking
### 1) Store page package
Create:
- one icon brief
- three thumbnail briefs
- title/tagline recommendation
- short description
- longer description paragraph set
- screenshot / caption guidance if helpful

### 2) In-game wording alignment
Make sure the release-facing language matches the real game:
- horror tone level
- co-op framing
- task / event language
- payout / progression / shop wording where relevant

### 3) Release notes
Draft a player-facing Sprint 4 / Milestone 4 update note that highlights:
- what exists now
- what changed since the earlier alpha
- what players should expect from the current build

### 4) Positioning guidance
Document:
- recommended genre / mood framing
- content-maturity disclosure notes
- words to avoid if they would overpromise the current scope

## Constraints
- no unsupported feature claims
- no giant lore dump
- no misleading “multiplayer chaos” promises that require more systems than the build has
- no art requests that assume a full outsourced asset pipeline
- keep copy readable on mobile

## Output format
Return:
1. concise summary
2. exact files changed
3. final store-page package elements
4. any dependencies or open questions
5. whether QA now has enough release-surface material
